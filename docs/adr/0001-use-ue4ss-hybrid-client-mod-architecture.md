# ADR-0001: Use UE4SS Hybrid Client Mod Architecture

## Status

Proposed

## Date

2026-07-14

## Context

本项目需要在《幻兽帕鲁》Steam PC 客户端中发现当前已加载的野生帕鲁和资源，读取可用属性，执行复杂筛选，并使用游戏内 UI 显示顶部引导线、方向标记和目标信息。

主要约束包括：

- v1 仅支持 Steam PC。
- Mod 必须纯客户端运行，不要求服务器安装组件。
- 支持单机和多人服务器客户端环境。
- 维护者首发仅验证单机；多人服务器属于未经保证的兼容目标，由社区测试和反馈补充证据。
- 真人玩家永远不能进入 ESP 候选或渲染结果。
- 中文为默认语言，并允许切换英文。
- 不修改存档、实体属性、概率或服务器状态。
- 数据来源限于客户端已加载、已复制或本地静态可知的信息。
- UI 需要支持多条件筛选，复杂度高于简单配置表单。
- 需要通过 Steam Workshop 分发并管理依赖。

UE4SS Lua 适合对象发现、Hook 和快速验证，但纯 Lua 不适合承担全部复杂 UI 和高频绘制。LogicMod/Blueprint Widget 能提供原生界面和绘制能力，但部分运行时对象和字段不一定暴露给 Blueprint。直接使用 C++ 可以获得更强控制和性能，但会显著提高首版开发、调试和版本维护成本。

## Decision

v1 采用 UE4SS Lua + LogicMod/Blueprint Widget 的混合纯客户端架构。

- UE4SS 是必需运行时依赖。
- Lua 层负责对象发现、Hook、对象生命周期、属性读取、标准化快照和筛选输入。
- LogicMod/Blueprint Widget 层负责 `Shift+E` 面板、中文/英文 UI、顶部引导线、屏幕外方向、标签和样式绘制。
- 通过自定义事件或明确的数据桥接接口在 Lua 与 Blueprint 之间通信。
- 所有实体必须在进入注册表前进行类型识别。
- `APalPlayerCharacter` 及等价真人玩家表示必须立即丢弃，不能进入后续适配、筛选或渲染。
- 未知或无法证明为非玩家的角色默认不显示。
- 使用事件驱动和增量缓存，避免每帧执行全量对象扫描。
- 对象发现、位置更新、属性更新和收集状态使用不同刷新周期。
- 使用统一 Widget 渲染预算，限制同时显示数量。
- PalSchema 和 Mod Config Menu 不作为强制依赖。
- Steam Workshop 包使用 Lua + LogicMods 类型，并声明 UE4SS 依赖。
- 只有在性能分析证明 Lua/Blueprint 热路径无法满足预算时，才把相关部分迁移到 UE4SS C++。

该决策不包含直接偏移读取、签名扫描、外部内存读取或网络层修改。这些方案如需评估，必须新增 ADR。

## Options Considered

### Option 1: Pure UE4SS Lua

优点：

- 原型速度快。
- 对象查找、Hook 和运行时实验方便。
- 开发工具链较轻。

缺点：

- 复杂筛选 UI 难以维护。
- 大量目标的高频投影和绘制可能产生性能问题。
- 容易形成回调、扫描和渲染逻辑混杂的单体脚本。

### Option 2: Pure LogicMod/Blueprint

优点：

- 原生 Widget 和 UI 开发体验好。
- 与游戏视觉风格集成方便。
- Steam Workshop 打包路径成熟。

缺点：

- 部分运行时类、字段和 Hook 不暴露给 Blueprint。
- 对对象发现和低层生命周期的控制不足。
- 调试复杂数据访问时可能受限。

### Option 3: UE4SS Lua + LogicMod/Blueprint Widget

优点：

- 同时获得运行时对象访问和原生 UI 能力。
- 符合 Palworld 社区已有实践。
- 可以先快速实现，再把明确的热点迁移到 C++。
- 容易把实体采集、筛选和渲染分层。

缺点：

- 需要维护 Lua 与 Blueprint 之间的接口。
- 生命周期和初始化顺序更复杂。
- 游戏更新可能同时影响脚本路径和 Widget 资产。

### Option 4: UE4SS C++ First

优点：

- 性能和控制能力最佳。
- 更适合大量实体和高级绘制。
- 可以建立更强的类型和测试边界。

缺点：

- 首版开发成本最高。
- 编译、ABI、SDK dump 和游戏版本兼容成本高。
- 在尚未证明产品假设前容易过度投入。

## Rationale

混合方案最适合当前阶段，因为项目同时需要复杂运行时数据访问和原生筛选界面，但尚未建立性能基线。纯 Lua 会把 UI 和高频绘制压力集中到脚本层，纯 Blueprint 又可能无法访问全部目标数据，C++ 首发则会提前承担较高维护成本。

该方案允许先用 Lua 在单机中验证对象分类和属性可用性，再通过 Widget 验证用户体验。多人复制边界由未来社区测试补充；只有经过测量确认的性能热点才迁移到 C++，避免把推测当成架构事实。

真人玩家在采集层排除是本决策的关键不变量。把玩家对象留到渲染层再隐藏，会让未来筛选器、调试输出、预设或 UI 缺陷有机会泄露玩家信息，因此不接受这种设计。

## Consequences

正面影响：

- 可以快速启动技术 Spike 和 MVP。
- UI 能使用游戏内 Widget，而不是调试控制台。
- 玩家排除边界清晰且可测试。
- 实体类别可以通过适配器逐步扩展。
- Steam Workshop 能管理 UE4SS 依赖和客户端安装。
- C++ 优化路径保留，但不阻塞早期验证。

负面影响：

- 需要定义并维护 Lua/Blueprint 数据桥接契约。
- 初始化、地图切换和 UObject 失效处理必须谨慎。
- Workshop、UE4SS、LogicMod 和游戏版本之间存在兼容矩阵。
- 客户端无法可靠获取服务器未复制的动态实体和属性。

需要后续注意：

- 不要把 50000m 筛选半径描述为 50000m 实体加载能力。
- 不要对缺失属性进行猜测或默认填值。
- 未知角色类型默认不显示。
- 普通树木和实例化植被不应进入早期实体扫描。
- 弱点显示需要独立验证稳定的数据来源。
- 直接内存读取无法解决服务器未发送数据的问题。

## Follow-ups

- [ ] 实现最小 Spike：发现一个野生帕鲁并显示顶部引导线。
- [ ] 维护者验证单机玩家类型排除；主机联机和专用服务器客户端等待社区测试，不作为首发阻塞项。
- [ ] 验证等级、性别、词条、IV、稀有状态和图鉴数量的数据路径。
- [ ] 定义 Lua 到 Widget 的最小数据契约。
- [ ] 建立扫描、投影、绘制和配置更新的性能基线。
- [ ] 确定默认最大显示数量与刷新周期。
- [ ] 在真实性能数据出现后决定是否需要 C++ 热路径。
- [ ] 若评估直接偏移、签名或外部内存读取，新增独立 ADR。
